# Отчёт о реализации MLOps-пайплайна для классификации деревьев по LiDAR-облакам точек

**Дата:** 22 апреля 2026  
**Автор:** opencode (assistant)  
**Проект:** LiDARTreeClassifier  
**Цель:** Реализация полного MLOps-пайплайна для классификации пород деревьев (Берёза, Ель, Сосна) по облакам точек LiDAR с использованием архитектуры PointNet, трекингом экспериментов через MLflow и конвертацией моделей в ONNX.

## Обзор проекта

Проект реализует классификацию пород деревьев по трёхмерным облакам точек, полученным с помощью LiDAR. Каждый образец содержит 4096 точек с координатами (x, y, z). Классы:
- 0 – Берёза (104 образца)
- 1 – Ель (242 образца)  
- 2 – Сосна (160 образцов)

Архитектура модели – PointNet (≈748 тыс. параметров) с T‑Net трансформерами для инвариантности к вращениям.

## Выполненные этапы

### 1. Подготовка данных
- Исходные данные `content/v2.h5` скопированы в `training/data/v2.h5`
- Создан файл `training/data/classes.json` с маппингом индексов классов на названия
- Реализована функция `load_dataset()` в `training/scripts/load_data.py` для загрузки данных

### 2. Аугментация данных
- Реализован модуль `training/scripts/augmentation.py`:
  - Случайный поворот вокруг оси Z
  - Случайное смещение (jitter)
  - Добавление гауссовского шума
- Поддержка множителя аугментации (`aug_factor`) для увеличения обучающей выборки

### 3. Архитектура PointNet
- Реализована в `training/models/pointnet.py`:
  - Класс `OrthogonalRegularizer` для ортогональной регуляризации T‑Net
  - Функции `conv_bn()`, `dense_bn()` для свёрточных и полносвязных блоков
  - Два T‑Net трансформера (для входных точек и признаков)
  - Глобальный max‑pooling для агрегации признаков
- Исправлена сериализация регулярзатора (добавлены `get_config()` и `from_config()`)

### 4. Настройка MLflow
- Создан `training/scripts/mlflow_setup.py` с функцией `setup_mlflow()`
- Локальное хранилище: `file://training/data/mlruns`
- Эксперимент: `LiDAR-Tree-PointNet`
- Логирование: параметры, метрики (поэпохно и финальные), confusion‑матрица, графики обучения

### 5. Пайплайн обучения
- `training/scripts/train.py` – функция `train_single()`:
  - Загрузка и аугментация данных
  - Создание TensorFlow Dataset
  - Компиляция модели (Adam/SGD, sparse categorical crossentropy)
  - EarlyStopping callback
  - Обучение с логированием в MLflow
  - Сохранение модели в `.keras` и логирование артефактов
- Поддержка параметров: epochs, batch_size, lr, optimizer, aug_factor, noise_std, dropout

### 6. Grid‑search (подготовлен, но не выполнен)
- `training/scripts/train_grid.py` – перебор гиперпараметров:
  - 432 комбинации (2×3×3×2×3×2×2)
  - Поиск лучшей модели по `val_acc`
- По решению пользователя grid‑search пропущен из‑за длительности выполнения на CPU

### 7. Экспорт в ONNX
- `training/scripts/export_onnx.py` – конвертация модели в ONNX
- Упрощённый скрипт `export_onnx_simple2.py` для обхода проблем с сериализацией
- Модель успешно конвертирована в `.onnx` (≈2.9 МБ)

### 8. Тестирование
- `tests/test_model.py` – проверка сборки модели и прямого прохода (тесты проходят)
- `tests/test_onnx_inference.py` – проверка инференса ONNX‑модели (пропускается, так как ONNX‑файл существует)

### 9. Автоматизация
- `Makefile` с командами:
  - `make train` – полный grid‑search
  - `make mlflow-ui` – запуск MLflow UI на localhost:5000
  - `make export-onnx RUN_ID=<id>` – экспорт модели в ONNX
  - `make clean` – удаление моделей и mlruns
- `docker-compose.yml` – конфигурация для запуска MLflow‑сервера с S3‑артефактами

## Проблемы и решения

1. **Ошибка сериализации OrthogonalRegularizer** – добавлены методы `get_config()` и `from_config()`.
2. **MLflow warning о deprecated file‑store** – рекомендуется переход на SQLite, но для локального использования файловое хранилище работает.
3. **Длительное обучение на CPU** – одна эпоха занимает ~60-90 секунд, полный grid‑search потребует много часов.
4. **Несбалансированные классы** – в данных преобладает класс "Ель" (242 образца против 104 "Берёза").
5. **Большой overfit gap** – требует увеличения количества эпох, регуляризации и аугментации.

## Рекомендации для дальнейшей работы

1. **Запуск grid‑search** с уменьшенным набором гиперпараметров для поиска оптимальной конфигурации.
2. **Увеличение эпох обучения** до 20-30 для сходимости модели.
3. **Использование аугментации** (`aug_factor=4-8`) для борьбы с переобучением.
4. **Балансировка классов** через взвешенную функцию потерь или oversampling.
5. **Интеграция с удалённым хранилищем артефактов** (S3) через MLflow.
6. **Создание REST API** для инференса ONNX‑модели.
